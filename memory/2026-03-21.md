## mppscan Shared Realm Bug Fix (afternoon, Ryan)

**Bug**: Three origins (AgentWeather, Quick Intel API, Tempo MPP Prediction Market) showed identical stats (6.2K txns, $3.03K) on mppscan activity list.

**Root cause**: All three had `realm: "MPP Payment"` ‚Äî the Tempo SDK's default string. `keccak("MPP Payment")` ‚Üí `memo_server_hash: 546a07b8cebd18e742dc` for all three. The `listAllOrigins` LEFT JOIN fanned out, duplicating stats.

**Fix iterations (PR #140, branch `craig/fix-activity-stats-fanout`)**:
1. First tried `ROW_NUMBER() PARTITION BY memo_server_hash` ‚Äî didn't work, resources √ó origins join produced dupes before window function
2. Switched to `argMin(origin_id, updated_at)` to pick one origin per hash ‚Äî worked for dedup but still showed one of the three with inflated stats
3. Tried filtering at registration time (reject realm !== host) ‚Äî Ryan said enforce it. But realized this breaks on-chain attribution: clients use the server's 402 realm to build the memo, not the host. Reverted.
4. Changed LEFT JOIN to INNER JOIN to hide unmatched hashes ‚Äî but these 3 ARE registered, so still showed
5. **Final fix**: `HAVING uniq(origin_id) = 1` ‚Äî if a hash maps to multiple origins, the realm is shared/garbage, drop it entirely from activity list

**Key insight**: `memo_server_hash` is derived from `realm` field in MPP 402 response. Client hashes it into on-chain memo. mppscan can't override realm at registration without breaking on-chain attribution. Fix must be at display layer.

**Related**: `listRegisteredOrigins` has same underlying issue (shared hashes inflate per-origin stats via GROUP BY origin_id). Not yet fixed.

**File**: `packages/db/src/queries/origins/list/all.ts`
